3.3 Data Preprocessing for Loan Risk Assessment

This section outlines the steps taken to prepare the loan dataset for modelling. Various techniques were applied to ensure data quality and compatibility with predictive algorithms.

3.3.1 Column Removal  
The first step involved removing irrelevant data from the dataset. The ApplicationDate column was dropped as it held no predictive value for risk assessment. This action reduced the feature count from 37 to 36 across the 20,000 records.  

Eliminating such non-informative elements streamlined the dataset. It ensured that subsequent processes focused only on attributes directly tied to financial behaviour and risk evaluation.

3.3.2 One-Hot Encoding of Categorical Variables  
Categorical features required transformation into a numerical format for model compatibility. OneHotEncoder with sparseoutput set to False was applied to EmploymentStatus, HomeOwnershipStatus, MaritalStatus, and LoanPurpose.  

This process converted each category into binary indicator columns, expanding the feature matrix. The dense array output facilitated integration with machine learning algorithms, capturing the distinct characteristics of each categorical value without implying any ordinal relationship.

3.3.3 Label Encoding of Ordinal Data  
For ordinal data, a different approach was adopted. LabelEncoder with default parameters was used on EducationLevel to assign integer codes to each unique category.  

This transformation resulted in a single integer column replacing the original text labels. It preserved the inherent order of education levels, making the data suitable for models that can interpret numerical sequences meaningfully.

3.4 Feature Selection with Recursive Feature Elimination

Feature selection was a critical step to refine the dataset for improved model performance. Recursive Feature Elimination, known as RFE, was implemented using RandomForestRegressor with the parameter nfeaturestoselect set to 10. This method iteratively removed the least important features based on their contribution to the model until only the top 10 remained. The selected features were AnnualIncome, LoanAmount, LoanDuration, DebtToIncomeRatio, TotalAssets, MonthlyIncome, NetWorth, InterestRate, TotalDebtToIncomeRatio, and RiskScore.

The outcome of this process narrowed down the dataset to the most influential variables. Visualisations were created to support the selection.

Description: This bar chart displays the importance scores of the 10 features chosen through RFE across the dataset. AnnualIncome and MonthlyIncome also rank high at 0.12 and 0.11 respectively. These values indicate that debt-related metrics and income stability are key drivers of risk assessment, guiding the focus for subsequent predictive modelling.

3.5 Feature Engineering for Enhanced Risk Indicators

To enrich the dataset with meaningful derived metrics, several new features were calculated. Six engineered variables were created: IncomeStabilityRatio as MonthlyIncome multiplied by 12 divided by AnnualIncome plus 1; NetWorthtoIncome as NetWorth divided by AnnualIncome plus 1; DebtStressIndex as DebtToIncomeRatio multiplied by InterestRate; CreditExperienceScore as CreditScore multiplied by the natural logarithm of LengthOfCreditHistory plus 1; HistoricalRiskFlag as the sum of BankruptcyHistory and PreviousLoanDefaults; and CompositeRiskIndex as 0.4 times 1 minus CreditScore divided by 850 plus 0.3 times DebtToIncomeRatio plus 0.2 times InterestRate plus 0.1 times HistoricalRiskFlag.

These derived features aimed to capture complex financial relationships. They provided deeper insights into applicant stability, debt burden, and historical risk patterns. Such metrics allowed models to assess risk beyond raw data, focusing on composite indicators that reflect real-world financial dynamics and applicant creditworthiness.

3.6 Data Scaling for Numerical Features

To ensure uniformity in feature magnitudes, scaling was applied to numerical data. StandardScaler was used on 13 continuous features including AnnualIncome, CreditScore, DebtToIncomeRatio, LengthOfCreditHistory, MonthlyIncome, NetWorth, InterestRate, TotalDebtToIncomeRatio, IncomeStabilityRatio, NetWorthtoIncome, DebtStressIndex, CreditExperienceScore, and CompositeRiskIndex. This method adjusted the data to a zero mean and unit variance. The fitted scaler was saved as scaler.pkl for future transformations.

Scaling prevented features with larger ranges from dominating the model learning process. It ensured equal contribution from all variables, improving the stability and performance of predictive algorithms during training and testing phases across the dataset of 20,000 records.

3.7 Train-Test Data Split

The dataset was divided into training and testing subsets to support model development and evaluation. The traintestsplit method was applied with testsize set to 0.2 and randomstate set to 42. This resulted in 16000 samples for training, representing 80% of the data, and 4000 samples for testing, accounting for the remaining 20%.

This split provided a robust foundation for model training on a majority of the data while reserving a significant portion for unbiased evaluation. Maintaining a consistent randomstate ensured reproducibility of the split, allowing consistent comparisons across different model experiments and configurations.

3.8 Model Training for Risk Prediction

This section describes the training process for three distinct models developed to predict loan risk scores. Each model was built with specific configurations to address the complexity of the task.

3.8.1 ElasticNet Regressor  
The first model trained was ElasticNet Regressor. It was configured with maxiter set to 10000 and randomstate set to 42. Hyperparameter tuning was conducted using RandomizedSearchCV with niter as 50, cv as 5, scoring based on r2, njobs as -1, and randomstate as 42. The best parameters identified were alpha as 0.0005336699231206312 and l1ratio as 0.9387755102040816.

This setup combined L1 and L2 regularisation to balance feature selection and coefficient shrinkage. The tuning process aimed to optimise predictive accuracy by exploring a wide range of parameter combinations, ensuring the model captured key patterns in the training data of 16000 samples without overfitting.

3.8.2 CNN Regressor  
The second model was a 1D Convolutional Neural Network, referred to as CNN Regressor, built using a sequential architecture. It included Conv1D with 32 filters, kernelsize as 2, activation as relu, and inputshape as 16 by 1; followed by BatchNormalization; then Conv1D with 64 filters, kernelsize as 2, activation as relu; another BatchNormalization; Flatten; Dense with 64 units, activation as relu; Dropout with rate as 0.3; and Dense with 1 unit for linear output. Training parameters were epochs as 50, batchsize as 32, validationsplit as 0.2, optimizer as Adam, loss as mse, and metrics including mae.

Its architecture featured sequential attention steps using entmax sparse masking across five decision steps with shared and step-specific fully connected layers.

3.9 LLM Integration for Underwriting Reports

To enhance decision-making with detailed explanations, a language model was integrated into the workflow. The OpenAI GPT-4o-mini API was used with a structured prompt to generate formal underwriting reports. Key parameters included temperature set to 0.7 for balanced creativity and maxtokens set to 500 to limit response length. The prompt outlined sections such as risk interpretation, financial strengths, risk drivers, and recommendations.

Input preparation involved loading the saved TabNet model, scaler, and threshold, then applying feature engineering on sample data to predict a risk score. The API returned a structured text report based on the score and business rules, which was displayed as part of the decision output. This integration bridged numerical predictions with narrative insights, aiding stakeholders in understanding risk assessments.

3.10 Web Application Development for Loan Risk Assessment

A web-based platform was developed to provide an interactive interface for loan risk evaluation. Streamlit was chosen as the framework, configured with default settings for rapid deployment. The user interface featured 10 numeric input fields for financial attributes like AnnualIncome, CreditScore, DebtToIncomeRatio, BankruptcyHistory, PreviousLoanDefaults, LengthOfCreditHistory, MonthlyIncome, NetWorth, InterestRate, and TotalDebtToIncomeRatio. Upon submission, the system processed inputs through feature engineering and scaling, predicted a risk score, applied a business rule flagging scores above 50 as high risk with a final decision of rejected, and generated an AI-driven report.

The output displayed the predicted risk score, model decision, business flag, final decision, and the full narrative report inline. This setup ensured accessibility for non-technical users to input data and receive comprehensive risk evaluations.

3.11 Testing of the Loan Risk Prediction System

To validate the system under different scenarios, testing was conducted with varied applicant profiles. Three test cases were designed: a high-risk applicant yielding a risk score of 79.53, resulting in rejection; a borderline applicant with a score of 51.55, also rejected due to the threshold of 50; and a low-risk applicant scoring 29.24, leading to approval.

These tests confirmed the system’s ability to handle diverse inputs and apply business rules consistently. Each case was processed through the full pipeline, from data input to final decision, ensuring that risk scores aligned with expected outcomes and governance policies were enforced accurately.

3.12 Challenges and Solutions in Development

Several technical hurdles emerged during the project’s execution. Handling the class imbalance in loan approvals, with 76% denials, posed a significant issue. Weighted evaluation metrics were adopted to address this, prioritising minority class performance without using oversampling methods like SMOTE, which were incompatible with regression targets. Another challenge was feature collinearity, especially between AnnualIncome and MonthlyIncome, as noted in research by Jemai and Zarrad (2023). Recursive Feature Elimination was applied to mitigate redundancy by selecting only the most impactful variables.

Model training with deep learning architectures also demanded high computational resources. Early stopping with patience set to 20 for TabNet and validation splits for CNN resolved overfitting risks, as supported by insights from Arik and Pfister (2021). These solutions ensured a balanced approach to data and model challenges.
