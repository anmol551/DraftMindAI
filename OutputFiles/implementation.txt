4.5 Models

4.5.1 Multinomial Naive Bayes

The process started by loading the resume dataset from Kaggle into a Pandas DataFrame with 2484 entries across four columns. This data underwent preprocessing that included removing unnecessary columns dropping ID and Resumehtml and expanding contractions using the contractions library. Text cleaning followed with lowercasing URL email and phone removal via regex and special character stripping along with whitespace normalisation. Tokenisation used NLTK wordtokenize to split text into words while removing stopwords but keeping 10 domain-specific keywords. Lemmatisation applied WordNetLemmatiser to refine tokens. Feature engineering then created TF-IDF vectors using TfidfVectorizer set with ngramrange of 1 to 2 maxfeatures at 5000 sublineartf enabled and mindf at 2. This vectoriser turned preprocessed text into a sparse matrix of shape 1987 rows by up to 5000 columns where each row represented a training sample.

The model relied on Scikit-learn's MultinomialNB class configured with alpha set to 0.1 for smoothing probabilities and preventing zero counts. Fitting occurred on the training set of 1987 samples split stratify based on labels ensuring proportional class representation. Input to the fit method included the TF-IDF sparse matrix and encoded labels from LabelEncoder applied to 24 job categories. During fitting the code incorporated sample weights calculated inversely proportional to class frequencies to manage imbalances. For instance if a class had fewer occurrences it received higher weights to balance the learning process. Edge cases such as words appearing less than the mindf threshold of 2 were automatically excluded from the vocabulary reducing noise in the matrix. The training process executed in a single pass as the model does not use epochs though it handled the input as a dense array internally for computation.

4.5.2 Bidirectional LSTM

The implementation began by deriving a custom vocabulary from the preprocessed tokenised and lemmatised text data which formed the basis for the Embedding layer. This vocabulary contained unique tokens from the dataset with the Embedding layer set to 128 dimensions to represent each token as a vector. The model architecture in PyTorch stacked an Embedding layer followed by a bidirectional LSTM layer with 128 hidden units two layers and dropout at 0.4 to capture forward and backward sequence dependencies. After the LSTM a Dropout layer with rate 0.4 reduced overfitting risks before a Linear layer mapped from 256 units to 24 output classes corresponding to the job categories.

Training employed the Adam optimiser with learning rate of 1e-3 and a StepLR scheduler that reduced the rate by gamma of 0.5 every 3 epochs. The setup processed 10 epochs with batch size of 32 and truncated sequences to a maximum of 256 tokens. Input sequences padded to this length ensured uniform processing while the CrossEntropyLoss function managed multi-class predictions. For edge cases the code included checks for sequences shorter than 256 tokens by padding them with zeros and skipped any batches with errors in tokenisation. This approach helped maintain consistent training flow across the 1987 samples.

4.5.3 DistilBERT

The implementation loaded the pre-trained distilbert-base-uncased model from the Transformers library and modified it for sequence classification with 24 labels. Input sequences first went through tokenisation using DistilBertTokenizerFast that set a maximum length of 256 tokens converting text into token IDs and attention masks. The model fine-tuning applied the AdamW optimiser with learning rate of 2e-5 and weight decay of 0.01 to regularise weights. A linear scheduler warmed up the learning rate over 10% of total steps before decaying it gradually.

Training ran for 10 epochs using batch size of 16 on the 1987 samples with the forward pass computing logits for the 24 classes. The CrossEntropyLoss calculated the loss based on these logits and true labels. Edge case handling involved monitoring for gradient explosions by clipping gradients and adjusting for any sequences exactly at the token limit. This ensured stable training without interruptions.

4.7 Challenges

During data preprocessing the process involved managing inconsistent text formats which complicated steps like URL and email removal using regex patterns from the 're' library. This step needed careful targeting of specific strings to avoid changing core content and thus prevent data loss for instance when dealing with embedded numbers or special symbols. Further text cleaning included lowercasing strings and expanding contractions with the 'contractions' library while tokenisation used NLTK's wordtokenize on texts up to 40000 characters requiring extra computational power. A custom stopword set also preserved 10 key resume terms during stopword removal and lemmatisation applied WordNetLemmatizer to tokens.

In model training class imbalance created issues especially with SMOTE using kneighbors set to 3 to generate synthetic samples from minority classes. This method needed smooth linking to the stratified 80% train-test split with 1987 training samples to keep balance. Setting up Bi-LSTM networks with dropout at 0.4 added more hurdles as it involved adjusting layer sizes such as 128 hidden units across two layers and batch processing to limit overfitting on uneven data. Other models like DistilBERT faced similar tuning demands for example with a learning rate of 2e-5 and 10 epochs which helped improve accuracy to 81.89% but required handling variable sequence lengths up to 256 tokens. Error checks in the code pipeline ensured that edge cases such as null values or duplicate entries did not disrupt feature engineering processes like TF-IDF vectorisation with ngramrange of 1 to 2.
